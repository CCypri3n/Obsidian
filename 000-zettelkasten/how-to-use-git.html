What is this?

This is a note containing useful information on how to use GIT.

Bash commands

Here’s a quick reference for common Git commands and their usage:

Note: Most Git commands are run in your project’s root directory. Replace <branch>, <file>, or <remote> with your actual branch name, file name, or remote name."

Terminal help (MacOS):

  • When using git log you may get into the pager "less" to display the log. To quit, press q (You may need to press Escape).
  • When editing a longer text, use :q (Vim) or ctrl+x (Nano), (You may need to press Escape).
Bash Arguments Description
git init opt: -b <branch> Create an empty Git repository or reinitialize an existing one.
-b: Specify the name of the first branch (Default: master)
git add <file names> Add file contents to the index.
git add . adds all files in the current directory and subdirectories.
git commit -m <message> Record changes to the repository.
-m: Add a commit message, for better clarity.
git push opt: -u origin <branch>; After you make and commit changes locally, you can share them with the remote repository using git push
-u origin <branch>: Useful when pushing a new branch, this creates an upstream tracking branch with a lasting relationship to your local branch (Only needed the first time you push).
git pull git pull, a combination of git fetch + git merge, updates some parts of your local repository with changes from the remote repository.
git branch <branch-name>: opt: --list; -d <branch> List, create or delete branches.
git branch <branch-name>: Create a new branch.
--list / None: Show all branches and the active one.
-d: Delete the specified branch.
git switch <branch> Switch branches.
switch -c <branch> will create and switch to a new branch.
git fetch <remote> When no remote is specified, by default the origin remote will be used, unless there’s an upstream branch configured for the current branch. It only downloads the latest changes from the remote but does not merge them into your current branch.
git clone <remote> Clone (download) a repository that already exists on GitHub or else, including all of the files, branches, and commits. <remote> should be a URL.
git status opt: -v; -s Most often used in its default form, this shows a good base of information.
-s: Give output in short format.
-v: Shows more "verbose" detail including the textual changes of any uncommitted files.
git log Browse and inspect the evolution of project files.
git remote -v Show the associated remote repositories and their stored name, like origin.
git merge <branch> Merge the specified branch into your current branch.
git stash Temporarily save changes that are not ready to commit.
git reset --hard / --soft / <commit> git reset <mode> <commit> moves the current branch pointer (HEAD) to the specified <commit>. The effect on your staged files (index) and working directory depends on the mode you choose. (See FAQ for more details)

Configure git

You can customize git to your needs using git config.
Use git config --list to see your current configuration.
When using git configyou can specify the level at which the change should occur:

  • System-wide (--system): Applies to all users on the system.
  • Global (--global): Applies to your user account, across all repositories on your machine.
  • Local (--local): Applies only to the specific repository where the command is run (default if no flag is provided)

This list explains the arguments you can add after git config <level>:

Argument Description
user.name <name> Sets your name for commits on the specified level.
user.email <e-mail> Sets your email for commits on the specified level.
core.editor <editor> Sets the editor for git on the specified level (e.g., vim, nano, or code --wait).
color.ui auto Enables colored output in the terminal.
alias.<short> <command> Creates a shortcut for a Git command (e.g., alias.co checkout lets you use git co instead of checkout).

FAQ

What is origin?

  • origin is an alias on your system for a particular remote repository. It's not actually a property of that repository.[1]
  • In Git, "origin" is a shorthand name for the remote repository that a project was originally cloned from. More precisely, it is used instead of that original repository's URL - and thereby makes referencing much easier.[2]

How do i use git reset?

Modes What happens to the index (staging area)? What happens to the working directory? Typical use cases
--soft Index is updated to match <commit> (all changes staged). Working directory is unchanged. Rewriting commits, keeping all changes staged for a new commit.
--mixed* Index is reset to match <commit> (unstaged changes). Working directory is unchanged. Undo last commit(s) but keep changes unstaged (default mode).
--hard Index is reset to match <commit> Working directory is reset to match <commit>. Completely discard all changes since <commit>, revert files too.

*--mixed is the default if no mode is specified.

Examples

git reset --soft <commit>

  • Moves HEAD to <commit>, keeps all changes from newer commits staged (ready to commit again), leaves your files as they are.

git reset --hard <commit>

  • Moves HEAD to <commit>, resets both the staging area and your files to exactly match that commit. All changes after <commit> are lost—use with caution.

What is the staging area?

The staging area (also called the "index") is an intermediate space between your working directory and your repository’s history. It lets you choose which changes you want to include in your next commit, allowing you to review and organize your work before saving it permanently. You add changes to the staging area with git add, and only staged changes are included in the next commit.

What is a repository?

A repository in Git is a directory that contains your project’s files and the entire history of their changes, managed by Git. It includes all your commits, branches, tags, and the Git metadata needed to track and manage your project’s evolution.

What is a remote?

A remote is a reference to a version of your repository that is hosted on another computer or server, often on a platform like GitHub or GitLab. Remotes let you collaborate with others by pushing your changes to, or pulling updates from, these external repositories. The default remote is usually named origin.
